**Create `validation/benchmark_parallel_scan.py`:**

Standalone micro-benchmark that measures sequential vs pscan likelihood evaluation time. This is Stage 1 of the two-stage benchmark design from the research doc.

Structure:
```python
"""Micro-benchmark: sequential lax.scan vs associative_scan for RLWM likelihoods.

Usage:
    python validation/benchmark_parallel_scan.py [--model MODEL] [--n-repeats N]

Outputs:
    - Console: timing table with speedup ratios
    - JSON: output/bayesian/pscan_benchmark.json
"""
```

Implementation:
1. Generate synthetic multi-block data matching real-data format:
   - 17 blocks (typical for main task), T=100 trials per block (MAX_TRIALS_PER_BLOCK)
   - Random stimuli, actions, rewards, set_sizes
   - Stacked format matching `*_multiblock_likelihood_stacked` input

2. For each model (M1-M6b), or just the model specified by `--model`:
   - Set typical MLE parameters (use reasonable values: alpha_pos=0.3, alpha_neg=0.2, phi=0.3, rho=0.7, capacity=4.0, kappa=0.1, epsilon=0.05, etc.)
   - JIT-compile both sequential and pscan variants
   - Warmup run (excludes compilation)
   - Time N repeats (default 20) of each variant using `time.perf_counter()` + `.block_until_ready()`
   - Compute speedup ratio = mean_sequential / mean_pscan
   - Verify numerical agreement (NLL difference < 1e-4)

3. Record results in `output/bayesian/pscan_benchmark.json`:
   ```json
   {
     "device": "cpu" or "NVIDIA A100-SXM4-80GB",
     "timestamp": "2026-04-14T...",
     "models": {
       "wmrl_m3": {
         "n_blocks": 17,
         "trials_per_block": 100,
         "n_repeats": 20,
         "seq_ms": 12.3,
         "pscan_ms": 8.7,
         "speedup": 1.41,
         "nll_agreement": 1.2e-6,
         "seq_compile_ms": 450.0,
         "pscan_compile_ms": 600.0
       }
     }
   }
   ```

4. Also measure compilation time (JIT first-call overhead) for both variants -- this matters for MCMC where the likelihood is called thousands of times but compiled once.

5. GPU memory measurement: if running on GPU, call `jax.devices()[0].memory_stats()` after each model run and record `peak_bytes_in_use`.

6. Print a summary table to console:
   ```
   Model       Sequential (ms)  PScan (ms)  Speedup  NLL Agree
   qlearning          3.2           2.8       1.14x   2.1e-7
   wmrl               8.1           5.3       1.53x   3.4e-6
   wmrl_m3           10.4           7.1       1.46x   1.2e-5
   ...
   ```

**Create `cluster/13_bayesian_pscan.slurm`:**

SLURM script for full GPU A100 benchmark (PSCAN-05 and PSCAN-06).

Base on `cluster/13_bayesian_gpu.slurm` (read for format). Key differences:
- `#SBATCH --job-name=bayesian_m3_pscan`
- `#SBATCH --time=12:00:00` (generous for first run)
- `#SBATCH --mem=48G`
- `#SBATCH --gres=gpu:a100:1`
- `#SBATCH --output=logs/bayesian_m3_pscan_%j.out`

Script body:
1. Activate conda environment (same as existing GPU scripts: try `rlwm_gpu` first, `ds_env` fallback)
2. Set JAX env vars: `JAX_COMPILATION_CACHE_DIR`, `XLA_FLAGS`, `NUMPYRO_HOST_DEVICE_COUNT=1`
3. Run Stage 1: `python validation/benchmark_parallel_scan.py --model wmrl_m3 --n-repeats 50`
4. Run Stage 2 (sequential baseline): `python scripts/fitting/fit_bayesian.py --model wmrl_m3 --chains 4 --warmup 1000 --samples 2000`
5. Run Stage 2 (pscan): `python scripts/fitting/fit_bayesian.py --model wmrl_m3 --use-pscan --chains 4 --warmup 1000 --samples 2000`
6. After both MCMC runs complete, the A/B comparison is done by comparing the two posterior NetCDF files. Add a small Python snippet at the end of the SLURM script that:
   - Loads both `wmrl_m3_posterior.nc` (sequential, from existing Phase 15 run or from Stage 2 sequential) and a `wmrl_m3_posterior_pscan.nc` (from Stage 2 pscan)
   - Compares group-level posterior means for all M3 parameters
   - Reports relative error and whether it's < 5% (PSCAN-06 criterion)
   - Compares WAIC/LOO and checks agreement to < 1.0 (PSCAN-06 criterion)
   - Writes comparison to `output/bayesian/pscan_ab_comparison.json`

**NOTE on pscan output file naming:** The `_pscan` suffix on the NetCDF filename is handled by Task 1's `save_results()` change. Task 2 assumes `wmrl_m3_posterior_pscan.nc` is produced when `--use-pscan` is passed.
