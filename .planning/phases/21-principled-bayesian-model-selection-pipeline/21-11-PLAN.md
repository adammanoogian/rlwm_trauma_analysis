<objective>
Build the 2-covariate L2 infrastructure for M3, M5, and M6a hierarchical models upfront, before winners are known.

Purpose: User-approved Option C — "the code for all model variants in principle, but the pipeline will only use one." Plan 21-07 dispatches each winner to the correct L2 refit path at runtime. M6b already has its full 4-covariate subscale variant (`wmrl_m6b_hierarchical_model_subscale`, 32 betas) and M1/M2 bypass L2 entirely. The missing piece is a 2-covariate hook (`lec_total` + `iesr_total`, both z-scored) for M3, M5, and M6a — wired alongside the existing `covariate_lec` parameter so backward compatibility holds for any caller that still passes a single covariate vector.

This plan covers:
- Extend M3, M5, M6a hierarchical models with an optional `covariate_iesr` kwarg (Approach A — see rationale below).
- Add a `beta_iesr_{target}` numpyro sample site (target = `kappa` for M3/M5, `kappa_s` for M6a) alongside the existing `beta_lec_{target}`.
- Apply both shifts on the probit scale before Phi_approx — the same pattern used for LEC-only.
- Add a helper `build_level2_design_matrix_2cov` in `level2_design.py` returning an (N, 2) z-scored design.
- Add pytest-based recovery + backward-compat tests (runs locally, no cluster needed).

Approach decision: **Approach A (additive parameter).** Rationale: Approach A adds a new optional keyword `covariate_iesr: jnp.ndarray | None = None` alongside the existing `covariate_lec`. Approach B (replace with a generic `covariate_matrix` + names list) would require rewriting all existing callers (`fit_bayesian._fit_stacked_model`, recovery tests, any Phase 16 notebooks, `21_run_prior_predictive.py`, `21_run_bayesian_recovery.py`, `21_fit_baseline.py`). Approach A preserves backward compatibility with zero changes outside the three model functions and the two test/helper files in this plan, keeping the blast radius tight. No other plan needs to be touched for infrastructure alone.

Output: Updated numpyro_models.py + a 2-cov design helper + a test suite. The pipeline code in plan 21-07 (Wave 6) then uses the new kwarg when the winner is M3/M5/M6a. The production SLURM pipeline runs `pytest scripts/fitting/tests/test_numpyro_models_2cov.py -v` as a local gate before submitting any cluster jobs (see plan 21-10 orchestrator update).
</objective>
