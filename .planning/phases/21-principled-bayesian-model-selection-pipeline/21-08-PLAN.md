  <action>
Create `scripts/21_scale_audit.py`.

**CLI:** `--l2-dir output/bayesian/21_l2/`, `--baseline-dir output/bayesian/21_baseline/`, `--winners-file output/bayesian/21_baseline/winners.txt`, `--hdi-prob 0.95`, `--fdr-alpha 0.05`, `--ess-drop-threshold 0.5`, `--output-dir output/bayesian/21_l2/`.

**Workflow:**

1. Read `winners.txt`; derive list of winner internal model ids.

2. For each winner:
   a. Load `{l2_dir}/{winner}_posterior.nc`. Skip + log warning if missing.
   b. Identify all `beta_*` variable names via `[v for v in idata.posterior.data_vars if v.startswith("beta_")]`. This enumeration is critical and deliberately pattern-based: for M3/M5/M6a winners it picks up BOTH `beta_lec_{target}` AND `beta_iesr_{target}` (2 sites total per winner, from the 2-cov L2 hook in plan 21-11). For M6b subscale winners it picks up all 32 `beta_{cov}_{param}` sites. For M1/M2 copy-through winners it returns an empty list. Log the enumeration explicitly: `print(f"[AUDIT] {winner}: {len(beta_sites)} beta sites: {beta_sites}")` to make the IES-R family inclusion observable in the log.
   c. For each beta site:
      - Extract flat posterior samples: `samples = idata.posterior[v].values.ravel()`.
      - Compute posterior mean, SD, 95% HDI via `az.hdi(samples, hdi_prob=0.95)`.
      - `excludes_zero_raw = (hdi_low > 0) or (hdi_high < 0)` (bool).
   d. Collect all beta sites across this winner into a DataFrame. Record the covariate family (`lec` / `iesr` / subscale-specific like `intr_resid` / `avd_resid`) as a column by parsing the site name; this makes the output table groupable by covariate family for the manuscript Table 3 generator in plan 21-10.

3. FDR-BH adjustment:
   - For each beta site, compute a surrogate p-value from the posterior tail probability: `p = 2 * min(mean(samples > 0), mean(samples < 0))`. This is the posterior two-sided tail probability.
   - Apply `statsmodels.stats.multitest.multipletests(pvals, alpha=args.fdr_alpha, method="fdr_bh")` and record the `excludes_zero_fdr` flag.
   - Important: FDR-BH on tail probabilities is NOT identical to frequentist FDR; use it as an HDI-agreement diagnostic. Document this caveat in the audit report.

4. Convergence diagnostics for beta sites specifically:
   - Run `az.summary(idata, var_names=beta_site_names)` → max R-hat, min ESS_bulk over the beta sites only.
   - Separately load `{baseline_dir}/{winner}_posterior.nc` and compute baseline min ESS on the NON-beta parameters (since baseline has no betas) as a reference.
   - Flag `ess_degraded` if min ESS on shared params dropped > 50% from baseline to L2 refit.

5. Per-winner artifacts:
   - `{output_dir}/{winner}_beta_hdi_table.csv`: columns `beta_site, covariate_family, target_parameter, posterior_mean, posterior_sd, hdi_low, hdi_high, tail_p, excludes_zero_hdi, excludes_zero_fdr`. `covariate_family` ∈ {`lec`, `iesr`, `iesr_intr_resid`, `iesr_avd_resid`}; `target_parameter` ∈ {`kappa`, `kappa_s`, `alpha_pos`, `alpha_neg`, `phi`, `rho`, `capacity`, `epsilon`, `phi_rl`} (parsed from beta site name).
   - Include a metadata row with `max_rhat_betas`, `min_ess_betas`, `ess_degraded`.

6. Aggregated output `{output_dir}/scale_audit_report.md`:
   - Header YAML-front-matter style block:
     ```
     ---
     pipeline_action: PROCEED_TO_AVERAGING  # or NULL_RESULT
     n_winners: <int>
     n_beta_sites_total: <int>
     n_hdi_excludes_zero: <int>
     n_fdr_excludes_zero: <int>
     ess_degraded_models: [<list>]
     ---
     ```
   - Per-winner sections listing notable effects (tail_p < 0.05 OR excludes_zero_hdi).

7. **Decision rule for pipeline_action + exit code protocol (plan-checker Issue #4 resolution):**
   - If across all winners `n_fdr_excludes_zero == 0`: pipeline_action=NULL_RESULT, **exit 0** (non-fatal; the YAML header pipeline_action tells 21_model_averaging.py to write `averaging_skipped.md` and exit cleanly, then 21.9 reports the null result).
   - Else: pipeline_action=PROCEED_TO_AVERAGING, **exit 0**.
   - Reserve **exit 1** for genuine audit errors ONLY: failed NetCDF load, import failure, unexpected ValueError. Do not conflate a valid null-result scientific finding with an audit error.
   - If any winner has `ess_degraded == True` OR new divergences appeared: add WARNING section to the YAML header (`ess_degraded_models: [...]`), still exit 0, but surface the degradation loudly in the markdown body.
   - This unified exit-0 semantics for PROCEED and NULL_RESULT lets the master orchestrator (plan 21-10) use `--dependency=afterok:$AUDIT2_JID` for the 21.8 job without the NULL_RESULT path failing the dependency — 21_model_averaging.py then reads the YAML header to decide whether to actually average or to soft-skip.

8. Import `arviz as az`, `pandas as pd`, `numpy as np`, `statsmodels.stats.multitest`. Guard the statsmodels import with a clear ImportError message if missing.

Follow CLAUDE.md conventions.

Invocation:
```
python scripts/21_scale_audit.py \
    --l2-dir output/bayesian/21_l2/ \
    --baseline-dir output/bayesian/21_baseline/ \
    --winners-file output/bayesian/21_baseline/winners.txt \
    --output-dir output/bayesian/21_l2/
EXIT_CODE=$?
echo "Scale audit exit: $EXIT_CODE"
# Unified exit-0 semantics: inspect YAML header of the audit report to
# distinguish PROCEED_TO_AVERAGING vs NULL_RESULT (both exit 0).
if [ $EXIT_CODE -eq 0 ]; then
    PIPELINE_ACTION=$(awk '/^---$/{flag=!flag;next} flag && /^pipeline_action:/{print $2; exit}' \
        output/bayesian/21_l2/scale_audit_report.md)
    echo "[AUDIT] pipeline_action = $PIPELINE_ACTION"
    case "$PIPELINE_ACTION" in
        PROCEED_TO_AVERAGING) echo "[PROCEED] Step 21.8 averaging enabled" ;;
        NULL_RESULT) echo "[NULL RESULT] Step 21.8 will self-skip; 21.9 reports null" ;;
        *) echo "[WARN] Unrecognized pipeline_action '$PIPELINE_ACTION'" ;;
    esac
else
    echo "[ERROR] Audit failed with exit code $EXIT_CODE — genuine audit error, not null result"
fi
source cluster/autopush.sh
exit $EXIT_CODE
```

Exit-code protocol (plan-checker Issue #4 resolution):
- Exit 0 + YAML `pipeline_action: PROCEED_TO_AVERAGING` → 21.8 runs and averages.
- Exit 0 + YAML `pipeline_action: NULL_RESULT` → 21.8 runs, reads the YAML header, writes `averaging_skipped.md`, exits 0, and 21.9 reports null.
- Exit 1 → genuine audit error (missing files, import failure); `afterok` dependency in plan 21-10 blocks 21.8 entirely.

The master orchestrator uses `--dependency=afterok:$AUDIT2_JID` for step 21.8. The NULL_RESULT branch does NOT block afterok since the exit code is 0; 21_model_averaging.py handles the soft-skip internally.
