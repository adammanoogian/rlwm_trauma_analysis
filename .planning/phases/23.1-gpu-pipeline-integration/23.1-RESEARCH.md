## 3. Decision Surface (per user's "do what's more correct in principle")

### Decision A: GPU architecture pinning

**Options:**
- **(a) Hard-coded `#SBATCH --constraint=l40s`** in every GPU SLURM script
- **(b) Env-overridable `#SBATCH --constraint=${GPU_TYPE:-l40s}`** with default L40S
- **(c) No constraint** + GPU-arch-suffixed cache path

**Recommendation: (b) env-overridable with L40S default.**
**Reasoning**: M6b proof was on L40S (m3g100 node). Hard-coding would lock to L40S forever — bad if Monash retires the partition or A100 becomes available. No-constraint risks cross-arch cache contamination (XLA HLO compiled for L40S may load on A100 with subtle correctness issues). Env-overridable preserves flexibility without giving up the safety default. Cache path follows pattern: `.jax_cache_21_baseline_gpu_${MODEL}` — JAX's internal hash already keys on backend (cpu/gpu) so explicit GPU-arch in path is overkill given the SLURM constraint.

### Decision B: Smoke-test cohort

**Confirmed by user: 10 participants × 100 warmup × 200 samples.**
**Reasoning**: Sufficient to exercise pmap + chain_method=parallel + LBA-isolation + cache write paths without burning hours. With `--chains 4 --gres=gpu:4`, expected per-model wall ~5-15 min. 5 models × 15 min = 75 min worst case (fits within `#SBATCH --time=2:00:00`). Loose convergence target (max_rhat ≤ 1.10) acknowledges small N.

### Decision C: Wave structure

**Confirmed by user: 3-wave structure.**
- Wave 1: Multi-GPU smoke validation (5 models) — must pass before any orchestrator changes
- Wave 2: Build 6 GPU SLURM variants + update orchestrator (parallel plans)
- Wave 3: Integration smoke (run pipeline at small N end-to-end via GPU orchestrator)

### Decision D: CLUSTER_GPU_LESSONS.md update

## 4. Operational Lessons (CLUSTER_GPU_LESSONS.md) — Must-Encode in Plans

### §2 Chain method (ALL plans must respect)
- `numpyro.set_host_device_count(N)` is CPU-only — **no-op on GPU**. Setting it doesn't change device count when `--gres=gpu:4`.
- For true pmap parallelism, need both `--gres=gpu:N` AND `chains=N` AND `n_devices >= chains`.
- On Windows/WSL, `chain_method="parallel"` silently falls back. **Smoke tests MUST run on Monash, not Windows dev box.**

### §5 JIT cache (must-have GPU-INT-04)
- Cache keyed on graph + dtype. Choice-only (float32) and M4 (float64) need separate dirs (already done).
- For Phase 21 GPU runs: `JAX_COMPILATION_CACHE_DIR=/scratch/${PROJECT}/${USER}/.jax_cache_21_baseline_gpu_${MODEL}` (per-model + `_gpu` suffix to coexist with future CPU runs at same `21_baseline` phase).
- `JAX_PERSISTENT_CACHE_MIN_ENTRY_SIZE_BYTES=0` to cache small compilations.
- **Lazy LBA import discipline**: Phase 21 must NEVER trigger any `lba_likelihood` import. Already enforced in `numpyro_models.py:wmrl_m4_hierarchical_model`. Smoke test should assert `jax.config.values["jax_enable_x64"] == False` post-fit.

### §6 SLURM templates (must-have GPU-INT-05)
- **`PYTHONUNBUFFERED=1`** in every script — without it, `[convergence-gate]` markers don't appear until job exits.
- **Per-model `timeout 600`** wrapper in smoke loop — Apr 16 incident (job 54845743) burned full 6h wall on qlearning alone because no per-model timeout. Fixed in `cluster/13_bayesian_fullybatched_smoke.slurm` — copy that pattern.
- **`source cluster/autopush.sh`** at end of every script — ensures logs land on origin/main automatically.

### §1 Trace-site collapse (already fixed, just don't regress)
- The 1000× qlearning speedup came from collapsing N=154 `numpyro.factor` sites into a single nested-vmap factor. Already in production via `wmrl_*_fully_batched_likelihood` functions. Smoke tests should verify nothing introduced a per-participant Python loop in any new code path.
